## 3. **System Design**

### **High-level Design vs. Low-level Design**
- **High-level Design**: Focuses on the architecture, components, and overall system interaction.
- **Low-level Design**: Detailed specifications, including algorithms, data structures, and class diagrams.

### **Design Patterns**
- **Common Patterns**:
  - **Singleton**: Ensures a class has only one instance.
  - **Factory**: Used for object creation without specifying the exact class.
  - **Observer**: A way of notifying all observers about an event.

### **Scalability, Availability, and Fault Tolerance**
- **Scalability**: Ability of the system to handle growing workloads.
- **Availability**: Ensuring the system is accessible when needed.
- **Fault Tolerance**: The system’s ability to continue operating correctly despite failures.

### **Database Sharding, Replication, and Indexing**
- **Sharding**: Distributing data across multiple databases.
- **Replication**: Duplicating data to improve availability and fault tolerance.
- **Indexing**: Speeding up query responses by organizing data in a searchable structure.

### **Load Balancing and Caching**
- **Load Balancing**: Distributing incoming network traffic across multiple servers.
- **Caching**: Storing data temporarily for faster access, reducing load on databases.

### **Microservices Architecture**
- **Microservices**: Designing systems with small, loosely coupled services that communicate via APIs.
- **API Gateway**: A server that acts as an API front-end, receiving API requests, aggregating the results, and passing them on to the backend services.
- **Service Discovery**: Mechanism to dynamically discover services in a distributed system.

### **CAP Theorem**
- **Consistency, Availability, and Partition tolerance**: In distributed systems, you can guarantee at most two of these three properties, but not all three at once.

## 10. **Cloud Technologies**

### **Key AWS Services**
- Core services like EC2 (compute), S3 (storage), RDS (databases), Lambda (serverless), and more.

### **Docker**
- A platform for containerization, enabling developers to package applications and their dependencies into isolated containers.

### **Kubernetes**
- An open-source platform for automating containerized applications' deployment, scaling, and management.

### **CI/CD Pipelines (Jenkins, GitHub Actions)**
- **Jenkins**: Automates the software development lifecycle (build, test, deploy).
- **GitHub Actions**: Automates workflows directly in GitHub.

### **Infrastructure as Code (Terraform, CloudFormation)**
- **Terraform**: Defines infrastructure with code to automate setup and management.
- **CloudFormation**: AWS service for modeling and setting up infrastructure using JSON or YAML templates.

### **Monitoring Tools (Prometheus, Grafana)**
- **Prometheus**: Open-source monitoring and alerting toolkit, primarily for time-series data.
- **Grafana**: Open-source data visualization platform, often used with Prometheus.

### **Cloud Cost Optimization Strategies**
- Leverage reserved instances, spot instances, right-sizing, and efficient use of cloud resources to optimize costs.
- **Security in Cloud**: Best practices for cloud security, including identity and access management (IAM), encryption, and multi-factor authentication.

### **Serverless Computing**
- **AWS Lambda**: A service that lets you run code without provisioning or managing servers. It automatically scales based on demand.

## 11. **Agile Methodologies**

### **Agile vs. Waterfall**
- **Agile**: Iterative, flexible approach with frequent releases and feedback.
- **Waterfall**: Sequential, rigid process with distinct phases.

### **Scrum Framework**
- Agile methodology that uses fixed-length iterations (sprints) to deliver features incrementally. Key roles include Scrum Master, Product Owner, and Development Team.

### **Kanban**
- A visual tool used to manage work, improve efficiency, and balance demand with capacity. It is more flexible than Scrum for continuous flow work.

### **Sprint Retrospective**
- A key Scrum event that focuses on improving processes. It’s a meeting where the team reflects on the sprint and identifies areas for improvement.

## 12. **Machine Learning Basics**

### **Types of Models**
- **Supervised Learning**: Model learns from labeled data (e.g., regression, classification).
- **Unsupervised Learning**: Model learns from unlabeled data (e.g., clustering).
- **Reinforcement Learning**: Model learns through trial and error (e.g., game playing).

### **Bias and Variance**
- **Bias**: Error due to overly simplistic assumptions in the model.
- **Variance**: Error due to model complexity, leading to sensitivity to fluctuations in training data.

### **Overfitting and Underfitting**
- **Overfitting**: Model is too complex, fitting noise rather than the underlying data patterns.
- **Underfitting**: Model is too simple, failing to capture important data relationships.

### **Regularization**
- Techniques like L1/L2 regularization are used to prevent overfitting by penalizing large coefficients in the model.

### **Evaluation Metrics**
- **Accuracy, Precision, Recall, F1 Score**: Common metrics to evaluate classification models.
- **Confusion Matrix**: Used to evaluate the performance of classification models.

### **Feature Engineering**
- The process of selecting, modifying, or creating features from raw data to improve model performance.

## 13. **Data Analytics Basics**

### **Steps in Data Analytics**
1. **Data Collection**: Gathering data from various sources.
2. **Data Cleaning**: Removing inconsistencies, handling missing data, and normalizing.
3. **Data Processing**: Transforming raw data into a usable format.
4. **Data Visualization**: Creating visual representations (e.g., charts, graphs) to better understand and communicate findings.

### **Data Modeling**
- **Regression Models, Decision Trees**: Techniques used to predict or classify data based on past observations.

### **Data Interpretation**
- Interpreting data analytics results to make informed decisions and derive actionable insights.
